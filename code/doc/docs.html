Audio Modules
Audio Modules are building blocks, which can be used to construct all kinds of musical instruments and / or audio effects. They are active real-time elements with inputs and outputs and an associated Prolog symbol for programmatic control. They can be constructed (or allocated), their inputs can be connected to outputs (of other modules) and destroyed (deallocated). Destroying a module requires two conditions: (a) the associated Prolog symbol has to be detached, and (b) module's output can not be connected to any other input. Once these two conditions are met, the audio module is automatically garbage-collected. To make the entire system work, a core module has to be constructed. To make the sound audible, some modules should be connected to the core's input. Core also functions as an audio source (connected directly to the sound hardware).
Common Behaviour:
[module]
 => 
Closes the module. The module is also destroyed and grabage collected if it is not connected to any input of any other module.

[module : *io]
 => 
Reports names of all inputs and outputs.

[module module]
 => 
Used for test purposes to update input signals and push the model one sample forward.

[module []]
 => 
Disconnect all inputs.

[module source]
 => 
Connects the first output of source to the first input of module.

[module DESTINATION source]
 => 
Connects the first output of source to the DESTINATION input of module.

[module DESTINATION source SOURCE]
 => 
Connects the SOURCE port of source to the DESTINATION input of module.

[module DESTINATION source []]
 => 
Disconnects the first output of source from the DESTINATION input of module.

[module DESTINATION source SOURCE []]
 => 
Disconnects the SOURCE port of source from the DESTINATION input of module.

Note: both DESTINATION and SOURCE can be either text or number.
[module *x]
 => 
Reports the current value of the first output port of the module.

[module SOURCE *x]
 => 
Reports the current value of the SOURCE output port of the module.

[module value]
 => 
Sets the value of the first input port of the module.

[module DESTINATION value]
 => 
Sets the value of the DESTINATION input port of the module.

Note: value can be either integer or double.
The commands above describe common behaviour of all audio modules.
Every module has a number of inputs and outputs, which can be connected. It should be noted that several output signals can be connected to one input. In such cases all those signals are added together. In fact, this is the simplest method of building a mixer.
[adsr module] => Creates an adsr module with linear raising and exponential falling slopes.
Note: module adheres to Common Behaviour.
Signals: [["TRIGGER" "ATTACK" "DECAY" "SUSTAIN" "RELEASE"] ["SIGNAL" "BUSY"]]
ADSR module. It works on a constant slope basis. This module should be connected directly to the GAIN input of oscillators, filters or amplifiers. For linear slopes use adsr-linear.
INPUTS:
Trigger: key on/off input. Range between 0.0 .. 1.0.
Attack: time. Range between 0.0 .. 16384.0, where 8192.0 = 1 second. Every step of 768 (6 * 128) multiplies time by 2.
Decay: time. Range same as above.
Sustain: level. Range between -16384.0 .. 0.0.
Release: time. Range same as above.
OUTPUTS:
Signal: output value. Range between 0.0 .. 1.0.
Busy: 0.0 = if finished release stage, 1.0 = otherwise.

[amplifier module] => Creates an amplifier module.
Note: module adheres to Common Behaviour.
Signals: [["ENTER" "GATEWAY"] ["SIGNAL"]]
This is an exponential audio amplifier. The range of "GATEWAY" control signal is -16384.0 .. 0.0. Every -1536 (-128 * 12) reduces the input signal by half.
[auto module data maximum_change] => Creates a sequencer connected to auto_data (maximum_change is optional).
Note: module adheres to Common Behaviour.
Signals: [["TRIGGER"] ["SIGNAL"]]
Plays back control sequences stored by the connected data module. The control changes can be ramped (filtered) if maximum_change was specified (ZERO is the default value = no ramping). The exact behaviour depends on the value of the "CONTROL" and "TRIGGER" signals of the auto data block.
trigger = 0 or control = 0 => inactive (no playback)
control = 1 => play once only
control = 2 => play looped
control >= 16 => inactive (no playback)
[auto_data module] => Creates a block holiding a sequence of control changes.
Note: module adheres to Common Behaviour.
Additional usage:
[module *value] => Returns the sequence of control changes.
[module []] => Erases all data.
[module [*value1 *time1] [*value2 *time2] ....] => Enters new data.
Otherwise defaults to common behaviour.
Signals: [["ENTER" "TRIGGER" "CONTROL"] []]
Auto data block holds control sequence as pairs of value and sample number after which the next value should be played. The source of the control values is the "ENTER" input of the data block.
A new sequence can be recorded by feeding 16 (or more) to the "CONTROL" input and then starting the recording by feeding 1 to the "TRIGGER" input. The recording stops when triggers disengages. However, recording may be paused by temporarily disengagin the control. Please, note, the recording should be stopped appropriately (by disengaging trigger while control is still 16).

[eg module] => Creates a 4-stage envelope generator module.
Note: module adheres to Common Behaviour.
Signals: [["TRIGGER" "TIME1" "TIME2" "TIME3" "TIME4" "LEVEL1" "LEVEL2" "LEVEL3" "LEVEL4"] ["SIGNAL" "BUSY"]]
Envelope Generator module. It works on a constant slope basis. I.e. the times are based on transitions across 16384.0 change between levels. As such, smaller differences between levels will cause the times to get shortened.
The slopes are linear and are indended to be used with exponential amplifier. For linear raising and exponential falling slopes use either eg-linear-expo or vca-eg module with integrated envelope generator.
Note: if retriggered before reaching level 4, it may start from a point between levels 4, 1, 2 or 3.
INPUTS:
Trigger: key on/off input. Range between 0.0 .. 1.0.
Time 1 - 4: range between 0.0 .. 16384.0, where 8192.0 = 1 second. Every step of 768 (6 * 128) multiplies time by 2.
Level 1 - 4: range between -8192.0 .. 8192.0 (however, can be more really).
OUTPUTS:
Signal: output value. Range between -16384.0 .. 0.0.
Busy: 0.0 = if reached level 4, 1.0 = otherwise.

[lfo module] => Creates an lfo module.

LFO generates low frequency oscillations, which can be used as control signals. Very useful for vibrato, tremolo, wah-wah effect and many more.

OUTPUTS:
Signal: output signal in range between -1.0 .. 1.0.
Negative: output signal in range between -1.0 .. 0.0.
Positive: output signal in range between 0.0 .. 1.0.

[morph module dimensions divisor] => Creates a morphing effect of several input values (optional: dimension integer [default = 2], divisor double [default = 1.0]).
Note: module adheres to Common Behaviour.
Signals: [["X" "A" "B"] ["SIGNAL"]] for morph 1
Signals: [["X" "Y" "A" "B" "C" "D"] ["SIGNAL"]] for morph 2
Signals: [["X" "Y" "Z" "A" "B" "C" "D" "E" "F" "G" "H"] ["SIGNAL"]] for morph 3
Creates a morphing effect. The effect can be one, two or three dimensional (default 2D for vector). The morphing formulas are:
SIGNAL = A * (1 - X) + B * X for morph 1.
SIGNAL = A * (1 - X) * (1 - Y) + B * X * (1 - Y) + C * (1 - X) * Y + D * X * Y for morph 2.
SIGNAL = A * (1 - X) * (1 - Y) * (1 - Z) + B * X * (1 - Y) * (1 - Z) + C * (1 - X) * Y * (1 - Z) + D * X * Y * (1 - Z) + E * (1 - X) * (1 - Y) * Z + F * X * (1 - Y) * Z + G * (1 - X) * Y * Z + H * X * Y * Z for morph 3.
The output signal can be (optionally) divided by divisor (if present and different than 1.0). INPUTS:
X, Y, Z: balance.
A, B, C, D, E, F, G, H: input signals.

[parameter_block module "style" maximum_change] => Creates a parameter block (parameter optional).
Note: module adheres to Common Behaviour.
Additional usage:
[module : *value] => Returns formatted value of the output if style was specified. Otherwise defaults to common behaviour.
Signals: [["SIGNAL"] ["SIGNAL"]]
Parameter block is a box holding a signal value. There are two types of parameter blocks:
Inactive: this is the default type. Its input is basically the same memory location as its output.
Active: actively copies its input to its output.
To make an active parameter block it is necessary to pass the maximum change parameter. It can be either integer or double and represents the maximum change of the value per sample. ZERO indicates exact copy (i.e. no ramping effect).
Parameter blocks can have their output values formatted into a text. To activate this feature a style parameter needs to be passed. It can have one of several pre-defined values: "index", "freq", "amp", "time", "onoff", "wave", "ratio" or "fm4algo".

Superstructure
The superstructure allows for constructing entire polyphonic music instruments, which can be connected to MIDI sources. It consists of a database of all modules and parameters, which can be automatically allocated, deallocated and connected to the core or other recipients of audio signals, such as oscilloscopes, spectroscopes and other effects. Additionally, the entire superstructure can be controlled from Command Centre Panel, which has an access to the database. The Command Centre is also a drop target (as well as any other visible panel) for Prolog scripts, which can automatically create new instrument and insert them into the superstructure.

A typical construction of an instrument and inserting it into the superstructure would require the following steps:
  • Create a new superstructure record by [Moonbase instrument Callback type line]. The instrument is basically the new superstructure record. Callback is necessary for the superstructure to be able to controll the new instrument (by sending MIDI messages to the callback). The type is important for automatic storing end re-storing of presets. The line is the mono or stereo output from the instrument and is necessary for automatic connections of audio signals.
  • Create a (moon)base, which will function as a callback for MIDI messages, eg. [(moon)base Callback]
  • Alternatively create arpeggiator or (poly-)sequencer. However, these modules will still need to have a (moon)base connected, eg. [arpeggiator Callback other-module]
  • Create and connect some global modules for the new instrument, such as volume, pan, delay, chorus, etc.
  • Make those modules accessible through the instrument's superstructure by calling [Insert module instrument : path]. Each module inserted this way will have several new parameter blocks constructed, connected to its inputs and made them accessible through the instrument's superstructure. Specialised parameter blocks have to be inserted individually using [InsertPB block instrument : path]. This method will not create any new parameter block.
  • In a loop create several polyphonic parts of the instrument. It will require creating a trigger module, which will need to be connected to the (moon)base callback. The purpose of the trigger is to: (a) receive from the Callback MIDI messages translated into signals, and (b) report back to the Callback if the part is still busy (through its busy input).
  • Also, in the same loop, part modules can be inserted into the superstructure. Please, note, the [Insert....] will not create duplicates for the same superstructure entries. Instead, the previously created parameter blocks will simply be connected to the inputs of modules from each new part.
  • After all parts are constructed, some more insertions can be made, especially for less significant modules (i.e. at the end of the superstructure).
  • MIDI controllers can be added to the Callback. It can be achieved by [InsertController ctrl-no optional-shift instrument : path]. This way the Callback is instructed to modify directly the input of the parameter block (specified by the path) in response to the MIDI control change message.
  • Some pre-set values can be set for some parameter blocks (if necessary) by [Lunar value instrument : path].
  • The entire superstructure can be connecte to the core (i.e. [ConnectAllMoons reactor]), or destroyed (i.e. [CloseAllMoons]). Individual instruments can be destroyed by [Moonbase instrument].

The purpose of the (moon)base module is to translate MIDI messages (or prolog commands) into controlling signals for triggers and parameter blocks (or more general audio modules). The (moon)base can be in either poly or mono mode.
In poly mode every new keyon message selects a trigger with its busy signal equal to zero. If no such trigger is available then it steals one of the busy triggers, most likely the one, which was busy for a while.
In mono mode all keyon messages are send to one selected (in fact the first one) trigger.
The MIDI control messages are translated into numeric values and applied to the first input of connected audio modules. In most cases these audio modules will be, in fact parameter block with just one single input. The value of a controller can be fractional.
Note:
1. If the index is greater than 31 and there is no connected audio module at index, then the value is treated as the lsb (i.e. value / 128) of a controller at index - 32.
2. If there is no audio module at index 126, then it is considered as mono switch.
3. If there is no audio module at index 127, then it is considered as poly switch.
4. Index 128 is also used by pitch MIDI command.

[core module centre_frequency headroom_fraction sampling_rate buffer_size output_device input_device capacity] => Creates a hardware connection.
[core module 330 0.4 48000 128 0 0 4096] => default values.
Note: module adheres to Common Behaviour.
Signals: [["MONO" "LEFT" "RIGHT"] ["MONO" "LEFT" "RIGHT"]]
Core creates an audio module, which connects directly to the underlying audio hardware. It can also record currently played audio to WAV files.
To create a hardware connection, it is necessary to provide at least a symbol. All numeric values (either integer or double) are optional. The headroom fraction, as it can only be double, can be ommited entirely or inserted at any position after centre frequency. The hardware connection can be closed in the same way as every other audio module.
Parameters:
module: a symbol representing audio module connected directly to the audio hardware.
centre_frequency: of middle E (Midi key number = 64). Can be either double or integer. Default = 330.0.
headroom_fraction: specifies the fraction of the entire dynamic range occupied by the range of audio signals -1.0 .. 1.0. For example, on a 16-bit dynamic range (-32768.0 .. 32768.0) a value of 0.25 would cause a sine wave operator to oscillate from -8192.0 .. 8192.0. Double. Default = 0.4.
sampling_rate: intended sampling frequency. Integer. Default = 48 kHz.
buffer_size: directly responsible for latency. Integer. Default = 128.
output_device: port number. Integer. Default = 0.
input_device: port number. Integer. Default = 0.
capacity: specifies the maximum number of expected modules, which can be active at the same time. Integer. Default = 4096.
To start audio recording it is necessary to provide at least its intended duration (in seconds). If the file name was not provided and the recording completes, then it will be lost. However, the recording can be stopped prematurely (by either [core] or [core "file_name.wav"]). If the recording is stopped and file name was not provided at the beginning, then it has to be provided at stopping (otherwise, the recording will be lost).

Opens MIDI interface. The default callback is the predefined income_midi command. It distributes incoming MIDI commands between instruments hooked to the superstructure.
The callback receives the following commands:
[income_midi keyoff channel key velocity] => Key Off
[income_midi keyon channel key velocity] => Key On
[income_midi polyaftertouch channel key value] => Polyphonic After Touch
[income_midi control channel value] => Control Change
[income_midi programchange channel program] => Program Change
[income_midi aftertouch channel value] => After Touch
[income_midi pitch channel lsb msb] => Pitch Change
[income_midi sysex .....] => System Exclusive
[income_midi timingclock] => Timing Clock
[income_midi START] => Start
[income_midi CONTINUE] => Continue
[income_midi STOP] => Stop
[income_midi activesensing] => Active Sensing

[AdsrPanel module attack decay sustain release] => Creates a panel for an ADSR module.
[module *x *y] => reposition on the screen
[module] => closes the panel
Creates a panel for 4 parameter blocks commonly associated with ADSR module. Range from 0 .. 16384 for times and -16384 .. 0 for sustain level. The resolution is 128 steps with left mouse button and 16384 with the right one.
Note: do not use AdsrPanel directly. Use BuildAdsrPanel instead.

[CommandCentre commander CCCB] => Create a command centre. CCCB parameter is optional and defaults to CCCB.
[commander *x *y] => reposition on the screen.
[commander] => closes the command centre panel.
CommandCentre panel enables effective control on all instruments, which are constructed as superstructures and are present on the list of all instruments. It should always be constructed with the predefined callback CCCB. It is possible to use custom callbacks, but its purpose would be rather dubious.
Note: you can drag-and-drop PROLOG batch files on the command centre.
[CorePanel module core reactor ConnectAllMoons CommandCentre commander] => Create a Core Panel. All parameters except module are optional (default values are: core, reactor, ConnectAllMoons, CommandCentre and commander).
[module *x *y] => reposition on the screen.
[module] => closes the command centre panel.
Core Panel controls audio hardware and audio recording. Note: you can drag-and-drop PROLOG batch files onto the control panel.
